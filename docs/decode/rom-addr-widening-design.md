# Microcode ROM address widening (+ ECP5 decoder measurement) — Design

**Date:** 2026-06-15
**Status:** Approved (brainstorm), pre-plan
**Branch:** `decode/rom-addr-widen`

## Goal

Lift the 256-entry microcode capacity ceiling that has constrained the last three
SH-4 privileged-arch milestones, by making the microcode address width
**auto-sized from the slot count** in the decoder generator. Separately, measure
whether switching the FPGA synth target from the combinational `direct` decoder
to the existing `rom` decoder reduces the J4 ECP5 area/timing that triggered the
PM2 `synth-size` Performance Alert.

These are **two independent levers** and this work treats them as such:
- **Capacity** → address widening (this is the committed deliverable).
- **FPGA slowdown** → a *measured* direct-vs-rom evaluation that we adopt only on
  evidence (FPGA-only, reversible).

## Background

### The decoder generator and the microcode model

The decoder VHDL is generated by `decode/gen-go/` (`cmd/cpugen`) from the TOML
spec under `decode/gen-go/spec/`. `make -C decode generate` produces the base
(J1/J2) decoder; `make -C decode generate-j4` applies the `spec/sh4` overlay for
J4. Base and J4 are **separate generation runs that overwrite the same
`decode/*.vhd` files**, and a given build uses one or the other — they are never
co-resident.

The microcode sequencer (`decode/decode_core.vhd`) holds an 8-bit `op.addr` that
**advances strictly +1 each non-stalled slot**. Each instruction occupies a
linear run of consecutive addresses; `predecode_rom_addr(opcode)` gives the start
address and the per-slot `dispatch` bit marks the last step. There is no step
counter, branch, or loopback. Consequently **every instruction's microcode steps
share one flat address space of 256 slots** — that is the capacity ceiling. PM2
hit 255/256 and had to fuse `SPC<-PC` into one slot to fit.

### The three decoder implementations

The generator emits three interchangeable table architectures (selected by VHDL
configuration):
- `decode_table_simple` — large combinational logic (3906 lines).
- `decode_table_direct` — QMC-minimized combinational logic over `op.code` +
  `op.addr` (992 lines). **This is what sim and all synth variants use today**
  (`cpu_decode_direct`). It grows with instruction/microcode count — the source
  of the PM2 J4 area growth (+89 generic cells / +558 ECP5 LUT4).
- `decode_table_rom` — a real `array(0 to 255) of std_logic_vector(74 downto 0)`
  ROM addressed by `op.addr` (505 lines, mostly the constant init). Its size is
  **constant regardless of instruction count**. It reads the microcode word on
  the **falling clock edge** by design ("so that xilinx uses a RAM ... falling
  edge to allow the ROM address to be computed") — i.e. it is engineered to
  infer block RAM. Only reachable via the FPGA-only `cpu_decode_rom_fpga` /
  `cpu_decode_rom` configs; CI (`pr-quick`) swaps `cpu_sim` to it and re-runs the
  180 µs ROM expecting 20 LED writes, so it is kept functionally correct (though
  CI tolerates "ROM decoder elaboration failed" by skipping — validated-when-it-
  builds, not a hard gate).

### Why `direct` is the default (and the ROM trade-offs)

`direct` is pure combinational and tech-independent — it is the **ASIC-portable**
default the project's identity rests on. The `rom` variant is an FPGA-shaped
trade: it moves microcode out of LUTs into BRAM (a real ECP5 LUT4 win) at the
cost of a clocked, falling-edge, half-cycle BRAM read and a hard dependency on
BRAM inference. On ASIC (no BRAM macro) the constant array re-synthesizes to
logic, so `rom` does **not** help the ASIC-nangate numbers and conflicts with the
portable default. Whether it improves *Fmax* on FPGA is unproven (half-cycle BRAM
access vs the combinational decode it replaces) — hence "measure, don't assume".

## Architecture of the change

### Lever 1 — Auto-size the microcode address width (the deliverable)

Today the address width is the literal `8` (`7 downto 0`) hardcoded in several
generator-emitted locations. The change: the generator computes a single width
per generation run and emits it everywhere consistently.

```
addr_bits = max(8, ceil(log2(num_slots)))
```

- The `max(8, …)` **floor preserves base/J1/J2 byte-identity**: base is ≤256
  slots, so it stays 8-bit and `make -C decode diff` remains green. The
  J1/J2-unaffected property (locked down in PM2) is preserved.
- **J4 auto-widens to 9-bit the moment its microcode exceeds 256 slots** — no
  flag, no `PRIV_ARCH` generic threading. The first change that needs >256
  (un-fusing PM2's `SPC<-PC`, or PM3's new entries) triggers the bump
  automatically. The ceiling is removed for good (works at 512, 1024, …).
- **Lazy bump (chosen):** J4 today is at 255/256, so it stays 8-bit until
  something adds a slot. We do not force an immediate jump to 9-bit; the smaller
  diff and preserved byte-identity-for-what-fits is preferred. (A floor bump for
  the J4 run to get 9-bit immediately is a one-line option if un-fusing lands in
  the same change and needs the room.)

**Emission points that must become width-driven** (currently hardcoded `8`/`255`/
`256`):
1. `decode_pkg.vhd` — `operation_t.addr : std_logic_vector(7 downto 0)`.
2. `decode_pkg.vhd` — `system_instr_addr_array` element `std_logic_vector(7 downto 0)`.
3. `decode_body.vhd` — `predecode_rom_addr` local `addr` and return width.
4. `decode_table_rom.vhd` — `array (0 to 255)` bound and the index expression.
5. `decode_table_direct.vhd` / `decode_table_simple.vhd` — the `op.addr` (`"s"`)
   seq-match bit count.
6. The generator's microcode-address assignment / capacity check (the `0..255` /
   256-entry logic in `internal/model/`), which both assigns addresses and is the
   place `addr_bits` is computed.

`decode_core.vhd`'s `op.addr := op.addr + 1` is already width-agnostic (follows
the `operation_t.addr` width) and needs no change.

### Lever 2 — ECP5 direct-vs-rom measurement (the evaluation)

Build J4 (`make -C decode generate-j4`) and run `yosys` `synth_ecp5` twice — once
with `cpu_decode_direct`, once with `cpu_decode_rom` — capturing LUT4 and EBR
(block-RAM) counts. `synth_ecp5` alone gives LUT4/EBR without nextpnr; Fmax (needs
nextpnr-ecp5, not available locally) is obtained from CI's existing `synth-ecp5`
job. Decision rule:

> Adopt `cpu_decode_rom` for the **FPGA** synth config only if it is a clear LUT4
> win **and** shows no Fmax regression in CI. Keep `cpu_decode_direct` as the
> portable default for sim and ASIC.

This is a synth-configuration choice (one config line per FPGA target), fully
reversible, and does not touch the RTL or the generator.

## Verification & testing

- **Generator unit tests** (`decode/gen-go/...`): add coverage that `addr_bits`
  is computed as `max(8, ceil(log2(slots)))` and that all emission points render
  the same width. Golden/byte-identity test for the base decoder must stay green.
- **Base byte-identity:** `make -C decode generate && make -C decode diff` → `OK`.
- **Functional sim (both decoders):** `decode/gen-go/regression.sh` (go test +
  sim + TAP) on base; and the `pr-quick` ROM-decoder swap path must still pass
  (20 LED writes) — extended to a 9-bit J4 build once J4 crosses 256 slots.
- **J4 round-trip suite** (`rte`-model + privmode/banktest/exctest/excguard) must
  stay green on a 9-bit J4 build when one is produced.
- **Measurement artifact:** record the direct-vs-rom ECP5 LUT4/EBR table in the
  PR description (and, if adopted, the Fmax delta from CI).

## Scope boundaries

- **In scope:** generator address-width auto-sizing; the emission-point edits; the
  ECP5 direct-vs-rom measurement; adopting `rom` for FPGA *iff* the data supports
  it.
- **Out of scope (separate work):** un-fusing PM2's `SPC<-PC` (an optimization the
  widening *enables* but does not require); PM3; reducing ASIC decode depth;
  wiring `PRIV_ARCH=true` into the J4 synth config (tracked separately — see the
  J4-synth-placeholder finding).

## Risks

- **Byte-identity regression** if the `max(8, …)` floor is dropped or a width
  point is missed — caught by `make -C decode diff` and the golden tests.
- **A missed emission point** producing a width-mismatched VHDL signal — caught by
  GHDL elaboration in the regression.
- **ROM elaboration fragility:** the `rom` decoder path has historically been
  allowed to soft-fail in CI; the measurement must confirm a clean 9-bit ROM
  elaboration before any adoption.
